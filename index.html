VPC Interface Endpoint · SQS (PrivateLink)

🔐 Private SQS Access from Private Subnet

Enable EC2 instances in private subnets to reach Amazon SQS without NAT Gateway, Internet Gateway, or public IP — using AWS PrivateLink and an Interface Endpoint.

Private DNS Traffic stays on AWS Interface Endpoint

Problem Statement

EC2‑B in a private subnet has no Internet access. It cannot reach the public SQS endpoint, so aws sqs send-message fails. We need private connectivity.

✨ LIVE TRAFFIC SIMULATOR ✨
✨ Step 1 — SSH to EC2‑A (bastion) from your laptop.
01
Prerequisites

VPC, public/private subnets, EC2-A (bastion), EC2-B (private), IAM role for SQS.

02
Create SQS Queue

Standard queue my-queue in ap-south-1.

03
IAM Role

Attach AmazonSQSFullAccess (or least‑privilege policy) to EC2‑B.

04
Security Group

SG for endpoint: allow HTTPS (443) from VPC CIDR.

05
Create Interface Endpoint

Service: com.amazonaws.ap-south-1.sqs, attach to private subnet, enable Private DNS.

06
Verification

SSH to EC2‑B, run aws sqs send-message → success!

Troubleshooting

❌ DNS resolves to public IP
Ensure Private DNS is enabled on the endpoint. Check with nslookup sqs.ap-south-1.amazonaws.com.
🔑 Timeout / Connection refused
Verify the endpoint security group allows inbound HTTPS (443) from the VPC CIDR. Also check that the route table for the private subnet has local routes; interface endpoints don't need a special route.
🛡️ AccessDenied
Confirm the IAM role attached to EC2‑B has the correct SQS permissions. If you use an endpoint policy, make sure it doesn't deny the action.
