SSH Chain & S3 Commands

# Connect from your laptop to EC2-A (bastion)
ssh -A ec2-user@<EC2-A-Public-IP>

# From EC2-A, SSH to EC2-B (private subnet)
ssh ec2-user@10.100.11.x

# Try S3 access BEFORE Gateway Endpoint (fails)
aws s3 ls s3://my-bucket/
aws s3 cp s3://my-bucket/test.txt ./
# Expected: timeout / error (no route to S3)
# AFTER creating Gateway Endpoint (route table auto-updated)
aws s3 ls s3://my-bucket/
aws s3 cp s3://my-bucket/test.txt ./
# Success! Traffic goes through Gateway Endpoint (AWS internal backbone)

Troubleshooting & Best Practices

❌ S3 access still failing after endpoint?
✔️ Verify the private route table contains the prefix list route (pl-xxxx) targeting the Gateway Endpoint.
✔️ Ensure EC2-B has the IAM role attached (S3 read permissions).
✔️ Check that you are using the same region for endpoint and S3 bucket.
✔️ If bucket policy exists, ensure it allows aws:sourceVpce condition.
🔑 SSH from EC2-A to EC2-B permission denied
➜ Use SSH agent forwarding (ssh -A) or copy the .pem key to EC2-A.
➜ Verify EC2-B security group allows SSH from VPC-A CIDR (10.100.0.0/16).
💰 Why Gateway Endpoint is cost‑effective?
Gateway Endpoints are free (no hourly charge). Replaces NAT Gateway for S3 access – saves ~$35/month per AZ plus data transfer costs. Perfect for private subnets that only need S3/DynamoDB.

Key Concepts Recap

Gateway Endpoint
Route table entry to S3 (prefix list), no public IP needed.
Prefix List
Automatically updates with S3 IP ranges – no manual CIDR maintenance.
No NAT required
Traffic never leaves AWS backbone, lower latency and cost.
IAM + Bucket Policy
Combine IAM roles and endpoint policies for defense in depth.
Zero internet egress
Private subnets stay private, no data exposure to public internet.
Clean‑up — Terminate EC2 instances, delete Gateway Endpoint, delete VPC and S3 bucket. Gateway Endpoint itself has no ongoing cost, but EC2 instances incur charges.
